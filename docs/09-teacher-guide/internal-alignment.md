# 킨더브레인 내부 정렬 — 투트랙 전략 (직원용 1페이지)

| 버전 | 날짜 | 변경 |
|---|---|---|
| v1.0 | 2026-07-17 | 최초 작성 — 직원 의견서(킨더브레인_의견정리) 논의과제 ④·①·데이터-3·4 대응, 투트랙 전략 반영 (사용자 승인) |

> **킨더브레인은 11월 킨더버스 오픈 이후부터 처음 학습하기 위한 시스템이 아니다.**

> 오픈 이후 실제 교사 데이터로 현장형 브레인으로 계속 성장한다.

## 1. 투트랙 — 무엇이 언제 일어나는가

| 구분 | 트랙 1 · 오픈 전 (지금~11월) | 트랙 2 · 오픈 후 (11월~) |
|---|---|---|
| 목적 | 초기 의도 이해 능력 구축 | 실사용 발화로 현장 적합화 |
| 대상 | 내부 직원·전문가·검수자 | 실제 교사 |
| 주 데이터 | 전문가 저작·직원 테스트·검수 문항·합성 뱅크 | 실제 교사 발화·화면 문맥·교사의 선택과 수정 |
| 검수 | 계획된 2인 검수 (kappa 또는 일치율) | AI 라벨 후보 + 운영 검수 → 같은 2인 게이트 |
| 주요 위험 | 합성 편향·내부 표현 편향 | 개인정보·실사용 분포 편향 |
| 산출물 | 선행 브레인 API (seed → v0.x) | 현장형 candidate brain |
| 평가 | KTIB·Arena | KTIB·Arena + 실사용 성공지표 |

두 트랙은 완전히 순차가 아니다 — **킨더버스 연결 준비(계약 동결·문맥 정의·실행 정책)는
트랙 1과 병행**한다. 상세: [two-track-launch-plan-v0.1](../06-runtime-integration/two-track-launch-plan-v0.1.md).

**데이터 역할은 섞이지 않는다** (물리적 강제 — DB CHECK):
합성 = TRAIN 후보 · 사람 검수 일치 = GOLD · 사람 저작+독립 검수 = 시험지(KTIB/HOLDOUT) ·
실서비스 = 11월 이후 별도 채널(PRODUCTION_SHADOW)로만 유입.

## 2. 성공지표 — "궁극 목표"와 "오픈 기준"은 다르다

- **궁극 목표(품질)**: KTIB 첫 시도 정확도 **96%** · 위험 의도 오발률(CWAR) ≤ **2%** ·
  위험 의도를 되묻기로 회피하지 않기(CCC) ≥ **80%** — 전부 `config/experiments.yaml` 단일 원천.
- **11월 오픈 기준(안전)**: 아래 9항목. *"모든 말을 완벽하게 알아듣는 브레인이 아니라,
  잘 아는 것은 빠르게 처리하고 모르는 것은 정직하게 묻는 브레인"*.

| # | 오픈 최소 기준 | 현재 장치 | 상태 |
|---|---|---|---|
| 1 | 주요 의도에 검수된 앵커(GOLD) 존재 | 2인 검수 승격 경로(즉석 문답→검수→GOLD) | 장치 ✓ · 데이터 채우는 중 |
| 2 | 시험지가 사람 저작·독립 검수 문항으로 구성 | KTIB 386문항(비합성·2인 O/X) + LLM 생성 금지 원칙 | ✓ |
| 3 | CRITICAL 7 위험 오판률이 기준 이내 | `cwar_max 0.02` + 위험 의도당 30문항 시험 | 장치 ✓ · Arena로 측정 |
| 4 | 애매하면 억지로 답하지 않음(clarify/abstain) | `clarify_margin 0.15`·`clarify_confidence 0.40`·`abstain_score 0.25` | ✓ |
| 5 | top 후보·confidence·margin 안정 반환 | infer 응답 계약(+risk_level·context_used, 2026-07-17 확장) | ✓ |
| 6 | 학습/시험 오염 검사 완료 | dedup 해시(split 미포함)·TRAIN 대조 거부·DB CHECK | ✓ 기계 강제 |
| 7 | 틀렸을 때 교정 구조 | 즉석 문답 교정 → evidence → 검수 → 승격 | ✓ 가동 중 |
| 8 | 응답·버전 기록 재현 가능 | inference_logs(request_id·단계 기록) + 시험지 4축 동결 | ✓ |
| 9 | 이전 브레인으로 롤백 가능 | brain_versions promote/reject — 이 문서 §5 런북 | 장치 ✓ · 런북 본 문서로 명문화 |

## 3. 틀렸을 때의 복구 구조 (정확도만큼 중요)

발화가 애매하다 → **되묻는다**(clarify, 회당 1회·세션 3회 상한) → 그래도 모르면 **모른다고
한다**(abstain — 위험 의도는 되묻기로 세탁 금지, CCC 80%) → 사람이 **즉석 문답으로
고쳐준다**(교정 evidence) → 2인 검수를 거쳐 GOLD → 새 후보 브레인 → **같은 시험지(Arena)로
증명된 경우에만 승격**. 수집은 실시간, 뇌의 변경은 통제된 승격뿐이다.

애매·교정·분류불능 발화는 대시보드 **INSIGHT·애매 발화** 카드에 출처(오픈 전 실험실 vs
오픈 후 실사용)별로 모인다 — "교사들이 실제로 어떻게 말하는가"의 정량 자료(CSV 내려받기 지원).

## 4. 의도별 적용 등급 (11월 적용 방식 — 개념)

오픈 시 모든 의도를 같은 수준으로 켜지 않는다. **의도마다** 시험 성적(Arena)과 위험
등급(리스크 모델)에 따라: **A(Assist·바로 연결) / B(Suggest·후보 제시 후 확인) /
C(Confirm·실행 전 명시 확인 — CRITICAL 7은 성적과 무관하게 항상 C) / HOLD(내부 추론만)**.
등급 산정 규칙·연결 준비 일정은 [two-track-launch-plan-v0.1](../06-runtime-integration/two-track-launch-plan-v0.1.md).
live 실행 코드는 그 문서의 게이트 통과 전까지 **HOLD**다.

## 5. 롤백 런북 (오픈 기준 #9)

1. 증상 확인: 특정 의도 오판 급증(대시보드·CWAR) 또는 승격 직후 회귀.
2. `brain_versions`에서 직전 promote 버전 확인 — 서빙 버전은 "최신 promote"라서
   **문제 버전을 reject로 기록하면 즉시 직전 버전으로 서빙이 돌아간다** (데이터 삭제 없음).
3. 원인 데이터는 삭제하지 않는다 — evidence는 남기고, 재검수 후 다음 후보에서 반영.
4. 사후: Arena 재실행으로 롤백 상태의 점수를 기록(감사 추적).

## 6. 리소스 (데이터-4 응답 — 실측 규모 기반)

모든 LLM 호출은 **토큰·비용이 건별 로깅**된다(`llm/client.py`). 실측 규모(2026-07 기준):
- 시험 채점 1회 = 문항 수 × 1호출 (현재 386문항) — 소형 모델 기준 회당 수백 원대(추정).
- 임베딩 전량(수천 短문) — 수백 원 미만(실측 규모).
- 합성 증산(campaign)만 대량 호출이며 품질 윈도 자동 중단이 있다.
원칙: **큰 지출(채점·증산)은 사람이 트리거**하고, 유입 0 재채점은 버튼이 스스로 막는다.

## 7. 직원 의견서 논의과제 4개에 대한 응답

| 논의과제 | 응답 |
|---|---|
| ① 가치 공감, 실행 방법 물음표 | §1 투트랙 + §4 의도별 등급이 "어떻게 쓰이는가"다 — 11월 첫날부터 자연어→기능 연결(등급별로 안전하게) |
| ② 데이터는 자산이자 리스크 | §1 데이터 역할 분리(물리 강제) + 2인 검수 게이트 + 오적재 방지(중복·오염 거부, 업로드 이력 원문 보관) |
| ③ 틀렸을 때 복구 구조 | §3 — 되묻기→정직한 abstain→즉석 문답 교정→검수→증명된 승격 |
| ④ 착수 전 내부 정렬 | 본 문서가 그 합의문이다 — 목적(§0 첫 문장)·대상(§1)·성공지표(§2)·데이터 표준(§1·아래 표준) |

**데이터 표준(요약)**: 모든 데이터는 출처 4필드(어느 채널로·누가 만들고·누구에 관한 것·어떤
행위자의 신호)를 갖는다. 워크플로 상태(label_state)와 품질(reliability_tier)은 별개 축.
시험지는 비합성+GOLD만(DB가 거부). 적대적 문항은 자연 분포와 분리 집계.

## 8. 삼각지대 실행 가이드 (직원 발견 혼동 — 관찰 포인트·목표 점검·기간 회고)

직원 테스트가 찾은 이 경계는 자동 가설(147개)에 2쌍이 없었다 — **사람이 자동화가 못 찾은
경계를 찾아낸 사례**다. 처리 순서(손으로 시스템에 심지 않는다 — 설계된 경로로만):

1. **최소대립 문항 작성**: 한 요소만 다른 쌍으로. 예) "민준이 *관찰 포인트* 잡아줘" ↔
   "이번 달 민준이 *목표 점검*해줘" ↔ "지난 학기 우리 반 *돌아보기* 정리해줘".
2. 두 검수자가 O/X 독립 판정 → **시험지 업로드**(그 경계를 재는 자가 생긴다).
3. 같은 발화들을 **즉석 문답**으로 교정 — 혼동 edge(GYM_CORRECTION)와 교정 evidence가
   설계된 경로로 자동 생성된다.
4. 채점(Arena) 실행 → 그 경계의 혼동이 실측되고, 개선 여부가 점수로 남는다.
5. 11월 후: 실제 교사도 같은 경계에서 헷갈리는지 INSIGHT 카드(출처 분리)로 확인 —
   실교사도 반복 혼동하면 온톨로지 경계 자체를 재논의한다.
